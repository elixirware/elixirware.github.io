The Central Thesis

The devices are already in the walls.

They're in the fields, the substations, and the factories. The only open question is whether the software inside them will be built to deny failure or to survive it. The manifesto's claim: the BEAM community spent four decades learning to survive it — and that operational discipline, not raw language preference, is what IoT is missing.

Let It Crash
Don't write defensive code for every failure. Isolate, crash, restart from a known-good state.
Process Isolation
Millions of lightweight processes with no shared memory — one sensor's bug can't corrupt another's state.
Operations as Architecture
Fleet success is measured in MTTR, update success rate, and ops headcount per device-year.
Partition Honesty
Edge sites must stay autonomous through WAN loss. The network failing is the normal case, not the exception.
What the Manifesto Argues

Six properties IoT needs.
The BEAM ships all six.

The manifesto maps each chronic IoT failure mode to a runtime property the BEAM has carried in production since the telecom era.

🌳
Supervision Trees
The hierarchy of supervisors and workers is simultaneously your fault-recovery plan, your boot order, and your architecture diagram. One artifact, three jobs — no drift between docs and reality.
💥
"Let It Crash" Philosophy
For a device you can't SSH into at 3 a.m., automatic restart from a known-good state beats heroic error handling. Crashes become routine, observable, and contained.
📡
Fleet Updates That Stick
Hot code upgrades and NervesHub-style OTA delivery shrink the vulnerability window between disclosure and patch — one of the manifesto's four headline operational metrics.
⏱️
Soft Real-Time Scheduling
Preemptive, per-process scheduling keeps latencies predictable under load — no single misbehaving handler can starve telemetry, control loops, or the local UI.
🔭
Observability by Default
Live introspection of any process on a running device, plus the Telemetry instrumentation standard, means diagnosing a field unit without a debugger cable — or a truck roll.
👥
Digital Twins as Processes
One supervised process per physical asset: a continuously synchronized twin with mailbox, state, and lifecycle. Session-per-asset is already precedented at massive scale on the BEAM.
The Tier Model

One thesis, four hardware realities.

The manifesto refuses one-size-fits-all. It classifies devices into resource tiers and prescribes a runtime per tier — including an honest "not the BEAM" where the VM doesn't fit.

T0

Where the Manifesto Says No

A movement that can't lose an argument can't win one.

The manifesto names its own boundaries, includes a go/no-go checklist for new hardware, and states what evidence would falsify it. An honest "no" is the manifesto working as intended.

Hard real-time control

The BEAM has no certified worst-case execution time. Motor control and protection logic run in C, Rust, or IEC 61131 on a dedicated core — with the BEAM supervising from outside the loop.

Below the VM floor

Tier-0 energy-harvesting and coin-cell devices can't host a VM at all. The honest prescription is native code, with the BEAM architecture mirrored at the gateway.

Raw distribution over the WAN

Erlang distribution is for trusted LANs. Fleet-to-cloud traffic at 10³–10⁸ devices goes through brokered MQTT with mutual TLS — partition-honest by design.

What would falsify it

Fleet-scale longitudinal evidence that BEAM deployments fail to outperform conventional stacks on MTTR, update success, vulnerability-window duration, and ops headcount — at comparable cost. The movement commits to publishing such evidence if it finds it.
